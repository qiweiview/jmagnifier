# 管理端“页面文案清理 + 弹框重构 + 交互体验修复” Spec

## 1. 背景

当前管理端已经具备登录、运行状态、映射配置、连接记录、报文记录五类核心页面，也补齐了部分弹框编辑和详情能力，但整体体验仍有明显的“展示感强、任务感弱”问题：

1. 页面里存在较多不服务任务的描述文字，既不传达业务状态，也不帮助用户决策。
2. 弹框虽然已经承担编辑、详情、清理等操作，但布局规则不统一，信息层级和操作优先级不稳定。
3. 交互路径更偏“把功能堆上去”，没有围绕用户的实际排障、配置、确认、回看流程来设计。

本 Spec 聚焦管理端前端本身，不改底层抓包和转发能力，目标是为后续 UI 调整提供一份统一的问题分析与重构方案。

## 2. 现状分析

### 2.1 当前代码位置

- 主页面框架：`src/main/resources/admin/index.html`
- 登录页：`src/main/resources/admin/login.html`
- 管理端逻辑：`src/main/resources/admin/assets/app.js`
- 管理端样式：`src/main/resources/admin/assets/style.css`

### 2.2 现状中的典型问题

#### 2.2.1 存在大量“无效描述文字”

当前页面和组件里有多处描述并不回答用户真正关心的问题，例如：

- `Adaptive Layered Simplicity`
- `Admin Access`
- `白蓝控制台`
- `白蓝简洁视图`
- `白蓝配色`
- `条件筛选`
- `详情弹层`

这些文字的共同问题是：

- 更像视觉主题、设计口号或实现方式说明，而不是任务文案；
- 不能帮助用户理解“我现在在哪、我能做什么、为什么要做、做了会怎样”；
- 在运维型系统里会稀释页面的可信度和专业感。

当前实现中，这类文案分散在多个入口：

- 侧边栏和全局页面头部来自 `app.js` 的 `pageHead()`；
- 登录页 Hero、标签和辅助信息写在 `login.html`；
- 运行状态页、连接记录页、报文记录页通过 `heroBadge()` 添加了装饰性标签；
- 侧边栏底部 `context-chip` 继续重复主题概念，而不是系统状态。

#### 2.2.2 页面说明文案没有围绕任务目标组织

页面副标题当前普遍偏泛，例如：

- `进程实时状态与抓包统计`
- `运行时转发规则管理`
- `已保存的 TCP 连接记录`
- `已捕获报文摘要`

这些描述虽然没有明显错误，但仍然停留在“对象是什么”，缺少“用户来这里要完成什么”。结果是：

- 首次进入页面时，用户缺少明确的操作预期；
- 复杂页面没有把重要能力突出出来；
- 筛选、编辑、排查这类高频动作没有被文案承接。

#### 2.2.3 弹框容器统一了，但弹框规范没有统一

当前所有弹框都复用同一个全局 Modal 容器，这是正确方向，但不同场景的布局和操作规范不一致：

- 映射编辑弹框是长表单，但没有做字段分组、步骤分段、必填/推荐说明；
- 连接详情弹框是“信息堆叠 + 表格附加块”，缺少下一步动作；
- 报文清理弹框同时放两个危险操作，层级相同，容易让用户直接做高风险动作；
- 报文详情弹框信息已经较完整，但仍残留中英混杂和技术表达不统一的问题，例如 `HTTP Start Line`。

更具体的问题包括：

- 标题统一性不足，有的按对象命名，有的按动作命名；
- 说明区有时缺失，有时过长，没有一致的“为什么要操作”表达；
- 页脚操作顺序不稳定，危险操作和取消操作的权重关系不清晰；
- 缺少明确的弹框类型定义：详情型、编辑型、确认型、危险操作型目前只是在样式类名上区分，没有产品规则。

#### 2.2.4 交互设计更多站在实现侧，而不是用户侧

几个典型表现：

1. 映射删除仍然使用浏览器原生 `window.confirm`，和全局弹框体系割裂。
2. 查询区域使用普通 `div` 拼装输入项，没有形成完整表单语义，回车提交、快速重置、格式提示都不足。
3. 连接详情里的“最近报文”只能看，不能继续跳到该报文详情或报文列表。
4. 报文清理只告诉用户“无法恢复”，但不告诉用户清理范围对应的影响规模。
5. 登录页占用大量空间展示品牌和视觉信息，但真正和登录有关的帮助信息很少。
6. 全局弹框只有开关，没有焦点管理、回到触发点、危险操作关闭保护等完整的可访问性规范。

#### 2.2.5 当前样式系统仍然偏“展示卡片”，不够“操作面板”

虽然此前已经做过信息密度优化，但整体视觉仍然保留较重的展示型写法：

- Hero 区块几乎每页都有，且默认带装饰性 `eyebrow`；
- 登录页左侧 Hero 占比很大，但任务只有“输入密码并进入”；
- 侧边栏顶部卡片和背景装饰信息量高，但状态信息量低；
- 弹框内常见的是一整块连续内容，而不是按用户任务拆成稳定的段落。

这会导致页面看起来“完整”，但操作时仍要额外思考。

## 3. 根因判断

当前问题不是单纯的“文案不好看”，而是三个层面的设计基线没有统一。

### 3.1 内容策略缺失

页面文案没有区分：

- 品牌识别文案
- 页面任务文案
- 状态反馈文案
- 风险确认文案
- 空态/错误态文案

于是视觉口号、英文标签、技术名词、实现术语被混放在同一层。

### 3.2 弹框缺少产品级类型约束

虽然代码里已经通过 `modalType` 和类名区分不同内容，但没有形成以下规则：

- 详情弹框看什么
- 编辑弹框改什么
- 确认弹框确认什么
- 危险弹框如何降低误操作

缺少规则后，新增弹框时容易继续按“能放下就行”的方式扩展。

### 3.3 页面组织更偏开发视角

很多页面是“把接口能力直接映射到页面”：

- 有列表就做表格；
- 有筛选就放输入框；
- 有对象详情就塞进弹框；
- 有操作就放按钮。

但用户真正的流程是：

- 巡检时快速判断是否正常；
- 配置时尽量少犯错；
- 排障时能顺藤摸瓜继续钻取；
- 做危险操作时能明确知道影响范围。

当前页面没有始终围绕这四类流程设计。

## 4. 目标

### 4.1 业务目标

- 清理页面中的无效描述文字，提升管理端的专业感和可信度。
- 统一弹框的结构、层级和风险表达方式。
- 让登录、巡检、配置、排障、清理这几类核心任务更顺手。

### 4.2 体验目标

- 用户进入任一页面后，第一眼能知道“这里主要做什么”。
- 弹框打开后，用户能快速判断“这是查看、编辑还是确认删除”。
- 高风险操作默认更谨慎，低风险高频操作默认更直接。
- 页面之间形成连续路径，而不是每次都重新找入口。

## 5. 非目标

- 本次不调整后端接口的数据模型。
- 本次不重做整个视觉风格，只在现有白蓝控制台风格内做收敛。
- 本次不引入复杂前端框架或组件库。
- 本次不扩展新的管理端业务模块。

## 6. 总体方案

本次优化拆成三条主线：

1. 页面内容治理：删掉无效文案，重写任务文案、状态文案、空态文案。
2. 弹框系统重构：建立统一的详情型、编辑型、确认型、危险型规范。
3. 交互链路修复：围绕登录、查询、编辑、钻取、清理这几条主流程补足用户体验。

## 7. 方案 A：页面文案治理

### 7.1 文案分层规则

所有前端文案必须落到以下五类之一：

1. 页面定位：说明“这个页面解决什么任务”。
2. 状态反馈：说明“系统当前怎样”。
3. 操作反馈：说明“刚刚的动作结果怎样”。
4. 风险提示：说明“继续操作会带来什么影响”。
5. 空态引导：说明“现在没有数据时下一步该做什么”。

不属于以上五类的文案，原则上不保留。

### 7.2 清理规则

以下内容应整体删除或替换：

| 当前文案 | 问题 | 处理建议 |
| --- | --- | --- |
| `Adaptive Layered Simplicity` | 口号式文本，不服务任务 | 删除 |
| `Admin Access` | 中英混杂，信息价值低 | 改为 `管理员登录` 或直接删除 |
| `白蓝控制台` | 只描述配色，不描述系统 | 删除 |
| `白蓝简洁视图` | 只描述视觉风格 | 删除 |
| `白蓝配色` | 与业务无关 | 删除 |
| `条件筛选` | 只是实现方式，不是价值 | 替换为具体筛选目的 |
| `详情弹层` | 只是组件形态，不是用户动作 | 删除 |

### 7.3 页面级文案重写建议

#### 7.3.1 登录页

目标不是展示品牌，而是尽快完成登录。

建议：

- 左侧 Hero 大幅收缩，保留产品名和一句用途说明即可。
- 主文案改为：
  - 标题：`登录管理台`
  - 说明：`输入管理密码后可查看运行状态、修改映射并排查连接与报文。`
- 增加轻量辅助说明：
  - `仅管理员可访问`
  - `登录失败时请检查当前服务端配置的管理密码`

#### 7.3.2 运行状态页

重点不是“白蓝简洁视图”，而是“快速确认系统是否健康”。

建议：

- 页面标题保持 `运行状态`
- 副标题改为：`查看映射运行情况、抓包写入状态和最近异常，适合巡检时快速确认系统健康度。`
- Hero Badge 改成真实信息，例如：
  - `每 3 秒自动刷新`
  - `共 X 条映射`
  - `当前 Y 条活跃连接`

#### 7.3.3 映射配置页

重点是“维护转发规则并减少配置错误”。

建议副标题改为：

- `新建、修改、启停转发映射。变更会立即影响对应监听端口和转发行为。`

#### 7.3.4 连接记录页

重点是“按条件定位连接，再继续下钻到报文”。

建议副标题改为：

- `按映射、客户端和时间范围筛选连接，查看生命周期并继续追踪相关报文。`

#### 7.3.5 报文记录页

重点是“查看报文内容并支持风险清理”。

建议副标题改为：

- `按连接、映射和时间范围筛选已捕获报文，查看内容预览并按范围清理历史数据。`

## 8. 方案 B：弹框系统重构

### 8.1 建立 4 类弹框模型

#### 8.1.1 详情型

适用：

- 连接详情
- 报文详情

规则：

- 标题用对象名：`连接 #123`、`报文 #456`
- 头部只放关闭操作
- 主体分为“摘要信息 + 关键内容 + 关联下一步”
- 不把次要字段挤进第一屏

#### 8.1.2 编辑型

适用：

- 新建映射
- 编辑映射

规则：

- 标题用动作名：`新建映射`、`编辑映射 #12`
- 头部放关闭，底部放 `保存` / `取消`
- 表单必须分组，至少拆为：
  - 基本信息
  - 监听配置
  - 转发配置
  - TLS 配置
  - HTTP 高级选项
- 对协议切换、TLS 开启等动态区域增加说明文字，而不是只做显示/隐藏

#### 8.1.3 确认型

适用：

- 删除映射
- 放弃未保存编辑

规则：

- 明确说明“确认什么”和“会发生什么”
- 主按钮文案不能只写 `确定`
- 优先写成：
  - `删除映射`
  - `放弃修改`
  - `继续关闭`

#### 8.1.4 危险型

适用：

- 清空全部报文
- 清空非当日报文

规则：

- 危险操作之间不能平级堆放
- 必须先解释范围，再给动作
- 默认强调影响较小的动作，影响最大的动作放在次级位置或二次确认后触发

### 8.2 弹框通用规范

#### 8.2.1 标题与说明

每个弹框都必须满足：

- 标题回答“当前是什么操作”
- 说明回答“为什么要做 / 做了会怎样”

不再允许只有标题、没有上下文解释的危险弹框。

#### 8.2.2 操作区顺序

统一规则：

- 主动作在右
- 取消在左或次级样式
- 危险动作使用明确文案，不使用抽象确认词

#### 8.2.3 关闭策略

统一规则：

- 编辑型且已修改：点击遮罩、右上角关闭、Esc 都要触发相同的“放弃修改确认”
- 提交中：禁止关闭
- 危险操作二次确认阶段：Esc 和遮罩关闭需要允许，但要保持状态可恢复

#### 8.2.4 可访问性

新增约束：

- 打开弹框后焦点进入标题后的首个可操作元素
- 关闭弹框后焦点返回触发按钮
- 详情型弹框支持键盘滚动和 Escape 关闭
- 危险型弹框需要 `aria-describedby` 指向风险说明

## 9. 方案 C：交互链路修复

### 9.1 登录链路

现状问题：

- 左侧视觉信息过多，输入区信息过少；
- 密码输入缺少明确校验提示；
- 登录失败文案过于粗糙。

改进建议：

- 登录页改为“轻品牌 + 强任务”布局；
- 密码框增加 `required`；
- 空密码直接前端拦截，提示 `请输入管理密码`；
- 登录失败区分：
  - `密码错误`
  - `登录服务不可用，请稍后重试`

### 9.2 查询链路

现状问题：

- 查询区域只是输入框堆叠，没有表单语义；
- 开始时间/结束时间没有格式示例；
- 无“重置筛选”；
- 回车行为不稳定。

改进建议：

- 连接记录、报文记录的筛选区改为真正的 `<form>`；
- 时间字段补充占位说明，例如 `2026-04-17 10:30:00`；
- 增加 `重置` 按钮；
- 回车默认触发查询；
- 查询中按钮进入 `查询中` 状态。

### 9.3 编辑链路

现状问题：

- 映射编辑字段很多，但没有按照心智模型分段；
- 切换协议或 TLS 后，新增字段直接出现，缺少引导；
- 删除仍走浏览器原生确认框，体验不统一。

改进建议：

- 映射编辑弹框分组并添加每组标题；
- 对以下动态区域给出辅助说明：
  - `监听协议`
  - `转发协议`
  - `监听 TLS`
  - `转发 TLS`
  - `HTTP 高级选项`
- 删除映射改为自定义确认弹框，统一到弹框系统内。

### 9.4 钻取链路

现状问题：

- 连接详情里的最近报文不能继续点击；
- 报文详情能查看内容，但缺少回到筛选结果和继续定位的上下文提示。

改进建议：

- 连接详情里的最近报文增加 `查看报文` 操作；
- 点击后直接打开对应报文详情；
- 报文详情头部增加来源上下文，例如：
  - `来自连接 #123`
  - `映射 #7`
- 保留关闭后回到原列表位置和筛选条件。

### 9.5 清理链路

现状问题：

- “清空非当日”和“清空全部”同级放置，风险权重不合理；
- 用户执行前不知道会删多少数据；
- 成功后只显示文本结果，没有明确说明当前列表已刷新。

改进建议：

- 清理弹框改成两段式：
  - 先选范围
  - 再确认执行
- 若后端可以提供预估值，则弹框先展示：
  - 将删除报文数
  - 将删除 payload 文件数
  - 保留日期
- 默认主推 `清空非当日`，`清空全部` 放入更谨慎的确认步骤。

## 10. 分页面方案

### 10.1 登录页

#### 问题

- 品牌区过大；
- 英文标签和主题词干扰任务；
- 登录帮助不足。

#### 方案

- 缩小左侧信息区，保留产品名和一句用途说明；
- 右侧登录框前置；
- 增加密码校验和失败提示；
- 删除无效标签。

### 10.2 运行状态页

#### 问题

- Hero 区装饰性信息多，状态引导弱；
- 指标虽然多，但缺少“异常优先”表达。

#### 方案

- Hero 只保留任务说明和真实状态标签；
- 把 `写入错误`、`失败`、`丢弃报文` 这类异常指标视觉上前置；
- 若存在异常，页面头部提供轻量告警摘要。

### 10.3 映射配置页

#### 问题

- 操作按钮虽已做状态驱动，但编辑表单仍然信息密集；
- 删除确认不统一；
- 字段说明不足。

#### 方案

- 编辑型弹框按配置域分组；
- 删除改为确认型弹框；
- 对风险字段补充说明，例如：
  - 修改监听端口会影响新的入站连接
  - 关闭证书校验仅建议测试环境使用

### 10.4 连接记录页

#### 问题

- 筛选说明不足；
- 详情弹框只能看，不能继续操作；
- 表格字段对排障链路支持一般。

#### 方案

- 筛选区增加重置和时间格式提示；
- 连接详情中把最近报文做成可下钻入口；
- 为状态异常的连接突出显示错误信息和关闭原因。

### 10.5 报文记录页

#### 问题

- 页面标题、副标题仍偏对象描述；
- 清理入口风险高但说明浅；
- 报文详情虽功能完整，但上下文回路还不够顺。

#### 方案

- 页面头部围绕“筛选、查看、清理”组织；
- 清理弹框改成分级确认；
- 报文详情头部增加来源上下文和返回语义；
- 中文术语统一，不再混用 `HTTP Start Line` 这类英文提示。

## 11. 实现建议

### 11.1 前端结构调整

建议优先抽象以下能力：

1. 页面头部元信息配置
2. 弹框类型配置
3. 统一的空态/错误态/成功态文案出口
4. 查询表单组件化拼装

具体到当前代码：

- `pageHead()` 从单一字符串拼接，升级为任务型页面头部渲染；
- `heroBadge()` 改成只承载真实状态，不再承载装饰性标签；
- `openModal()` / `closeModal()` 增加焦点管理和关闭策略；
- 删除映射增加新的确认型弹框，不再使用原生 `confirm`；
- 查询区从 `div.toolbar` 改为表单式结构。

### 11.2 样式层调整

建议新增而不是继续堆叠现有类名：

- `page-intro`
- `page-context-meta`
- `dialog-description`
- `dialog-section`
- `dialog-danger-summary`
- `filter-form`
- `empty-state`

目标不是做更多样式，而是把“页面说明”“弹框说明”“风险说明”“筛选区域”从视觉上分开。

### 11.3 文案来源治理

建议把页面级文案集中收口到单独的配置对象，避免继续散落在：

- HTML 静态模板
- `pageHead()` 调用参数
- 局部按钮拼接
- 空态和错误态字符串

这样后续才能持续收敛。

## 12. 落地顺序

建议按以下顺序实施：

1. 先清理全局无效文案，统一页面副标题和空态文案。
2. 再重构弹框系统规则，优先处理删除映射和报文清理两个高风险场景。
3. 然后补齐登录页、查询区、连接到报文的钻取链路。
4. 最后做细节打磨，包括焦点管理、键盘行为、反馈一致性。

## 13. 验收标准

### 13.1 文案

- 页面中不再出现与任务无关的主题口号、配色标签、实现术语标签。
- 所有页面副标题都明确说明页面目标或操作价值。
- 危险操作文案明确表达影响范围和不可恢复性。

### 13.2 弹框

- 所有新增或改造后的弹框都能归类到详情型、编辑型、确认型、危险型之一。
- 删除映射不再使用浏览器原生确认框。
- 危险操作不再把多个高风险动作等权并列展示。

### 13.3 交互

- 登录页支持空密码前端校验。
- 查询区支持回车查询和一键重置。
- 连接详情可以继续进入报文详情。
- 弹框关闭策略、按钮顺序、成功/失败反馈保持一致。

## 14. 风险与注意事项

- 文案清理容易被误做成“单纯删字”，但核心是改成任务型表达，不能只删除不补充。
- 弹框规范如果只改样式、不改交互规则，问题会继续重复出现。
- 危险操作优化如果没有后端预估数据，至少也要在文案上明确范围定义，避免用户盲删。
- 登录页收缩视觉区时，要控制好品牌识别的最低保留量，不必完全做成工具化裸页。

## 15. 结论

当前管理端的主要问题不是能力不足，而是表达方式和交互规则没有真正围绕用户任务组织。下一步不应继续零散修按钮和文案，而应先建立统一的内容策略、弹框模型和交互基线，再按页面逐步落地。
