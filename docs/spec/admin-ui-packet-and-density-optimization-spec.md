# 管理端“报文记录”与全局信息密度优化 Spec

## 1. 背景

当前管理端已经具备运行状态、映射配置、连接记录、报文记录四个主模块，但在日常排障场景里有两个明显问题：

1. “报文记录”详情页只提供并排的文本预览与十六进制预览，缺少更符合排障习惯的单区域切换、快捷复制，以及针对 HTTP JSON 请求体的结构化阅读能力。
2. 整体页面字号、控件高度、卡片留白偏大，在常见 1440px 宽度或笔记本屏幕下，同屏可见信息量偏少，列表与详情阅读效率较低。

本 Spec 聚焦这两个问题，目标是在不改变产品核心结构的前提下，提升报文阅读效率和全站信息密度。

## 2. 现状分析

### 2.1 报文记录模块现状

- 报文列表页支持筛选、分页、详情弹层。
- 报文详情弹层当前结构为：
  - 基础元信息网格；
  - “下载负载”按钮；
  - 左右双栏预览：`文本` + `十六进制`。
- 当前详情接口 `/api/packets/{id}` 返回的关键字段包括：
  - `contentType`
  - `protocolFamily`
  - `applicationProtocol`
  - `textPreview`
  - `hexPreview`
  - `previewBytes`
  - `previewTruncated`
- 当前 `textPreview` 是将 payload 按 UTF-8 转文本后，把控制字符替换为 `.`。这会破坏 HTTP 报文原始换行结构，前端无法可靠区分请求行、Header、Body，也无法稳定识别并格式化 JSON Body。

### 2.2 页面密度现状

当前样式整体偏“展示型”，不够“控制台型”：

- 主按钮最小高度 `42px`，输入框最小高度 `44px`。
- 主导航项最小高度 `54px`。
- 顶部栏最小高度 `72px`。
- 页面 Hero 标题最大可达 `40px`，登录页标题最大可达 `52px`。
- 表格单元格 `padding: 14px 14px`，卡片与面板圆角、内边距普遍较大。

结果是：

- 表格一屏能看到的记录数偏少；
- 筛选条占据纵向空间较多；
- 弹层详情可视区域有限，报文主体被压缩；
- 导航与装饰性内容占比偏高。

## 3. 目标

### 3.1 业务目标

- 降低查看单条报文的操作成本。
- 提升 HTTP 报文特别是 JSON 请求体的可读性。
- 提升同屏信息承载量，适配长时间监控与排障。

### 3.2 体验目标

- 报文主体阅读路径更直接：进入详情后聚焦“内容”而不是被双栏挤压。
- 对可识别的 HTTP JSON 内容提供“开箱即用”的格式化能力。
- 常见操作支持一键完成：复制当前视图、复制请求体、下载原始负载。
- 页面整体更紧凑，但不牺牲可点击性和可读性。

## 4. 非目标

- 不重做导航结构与页面信息架构。
- 不引入复杂代码编辑器组件。
- 不在本次需求中支持 XML、protobuf、gzip 解压、chunked 重组等高级协议解析。
- 不修改抓包模型本身的存储策略，只补足展示所需的预览/解析能力。

## 5. 方案总览

本次优化拆成两个子项：

1. 报文记录详情阅读体验升级。
2. 全站视觉密度与布局尺度收敛。

建议按两个迭代落地，但共享一套设计原则：减少无效留白，保留清晰层级，优先服务排障效率。

## 6. 子项 A：报文记录详情升级

### 6.1 用户场景

#### 场景 A：快速查看单条报文

用户在报文列表中点击“详情”，希望优先看到：

- 这条报文是什么协议、请求还是响应、大小是否截断；
- 报文主体；
- 可以立即复制或切换查看方式。

#### 场景 B：查看 HTTP JSON 请求体

用户确认这是 HTTP 请求，希望快速判断：

- 请求行与 Header；
- Body 是否为 JSON；
- JSON 是否能自动格式化；
- 是否可以直接复制格式化结果发给其他同事。

### 6.2 交互设计

#### 6.2.1 详情弹层结构调整

详情弹层改为三段式：

1. 头部摘要区
2. 内容工具栏
3. 报文主体区

#### 6.2.2 头部摘要区

保留现有元信息，但收敛为更紧凑的摘要布局，优先展示：

- 报文 ID
- 方向
- 协议族 / 应用协议
- Content-Type
- HTTP Method / URI / Status
- 原始大小 / 捕获大小 / 是否截断
- 接收时间

不常用的网络地址信息继续保留，但放入次级信息区，避免主体区域被压缩。

#### 6.2.3 主体查看方式

报文主体使用同一区域 tab 切换，而不是左右双栏并列。

一级 Tab：

- `文本`
- `十六进制`

默认进入 `文本`。

理由：

- 单区域阅读宽度更大，更适合长 Header、长 JSON、长十六进制内容；
- 能为复制、格式化、状态提示预留统一工具栏；
- 移动端无需额外重排。

#### 6.2.4 文本视图内的 JSON 增强

当满足以下条件时，在 `文本` Tab 内显示二级切换：

- `原文`
- `JSON 格式化`

启用条件：

- 明确为 HTTP 报文；
- 能识别出 Body；
- `Content-Type` 为 `application/json` 或 `application/*+json`；
- Body 可成功解析为 JSON；
- 当前预览未因截断导致 JSON 不完整。

默认规则：

- 文本 Tab 默认打开 `原文`；
- 当检测到可格式化 JSON 时，显示 `JSON 格式化` 切换；
- 用户切换后仅影响当前详情弹层，不写入全局状态。

#### 6.2.5 快捷复制

内容工具栏提供以下动作：

- `复制当前视图`
- `复制请求体`
- `下载原始负载`

复制规则：

- 当前位于 `文本 > 原文`：复制原文预览内容；
- 当前位于 `文本 > JSON 格式化`：复制格式化后的 JSON；
- 当前位于 `十六进制`：复制当前十六进制展示内容；
- `复制请求体` 仅在 HTTP 且可识别 Body 时显示；
- 当预览被截断时，复制行为仅复制当前预览，并明确提示“内容已截断”。

复制反馈：

- 成功：按钮短暂切换为 `已复制`
- 失败：显示轻量错误提示，例如 `复制失败，请重试`

### 6.3 界面规格

#### 6.3.1 弹层布局

- 弹层宽度建议从当前 `1040px` 提升到 `min(1280px, 96vw)`。
- 弹层内容高度维持可滚动，但主体区优先保证可视高度。
- 元信息区与主体区之间使用更轻量的分隔，不再堆叠多个大块卡片。

#### 6.3.2 主体区样式

- 主体区使用单一深色代码阅读面板。
- Tab、二级切换、复制按钮放在同一工具栏。
- 支持横向滚动，避免长 Header 或长 JSON 被强制折行后难以阅读。
- 文本默认保留换行；十六进制按 16 字节或 32 字节分行展示。

#### 6.3.3 状态提示

在主体区顶部显示非阻断提示：

- `仅展示前 N 字节`
- `当前报文已截断，JSON 格式化不可用`
- `未识别到 HTTP Body`
- `当前内容不是 JSON`

### 6.4 后端能力要求

当前 `textPreview` 仅适合粗略浏览，不足以支持 JSON 格式化，因此需要扩展详情接口。

#### 6.4.1 接口

继续使用现有接口：

- `GET /api/packets/{id}`

在响应中新增 `payloadView` 对象，建议字段如下：

```json
{
  "payloadView": {
    "textRaw": "原始文本预览，保留换行",
    "hex": "十六进制预览",
    "previewBytes": 1024,
    "previewTruncated": false,
    "http": {
      "isHttp": true,
      "startLine": "POST /api/demo HTTP/1.1",
      "headersText": "Host: ...\\r\\nContent-Type: application/json",
      "bodyText": "{\"name\":\"demo\"}",
      "bodyDetected": true,
      "bodyJson": true,
      "bodyJsonPretty": "{\\n  \"name\": \"demo\"\\n}",
      "bodyTruncated": false
    }
  }
}
```

说明：

- `textRaw` 用于文本原文视图，必须保留原始换行，不再将控制字符统一替换为 `.`。
- `http` 为可选对象，仅在明确识别 HTTP 时返回。
- `bodyJsonPretty` 由后端生成，前端只负责展示与复制，避免前端基于不完整预览做不稳定解析。

#### 6.4.2 HTTP Body 解析规则

对于 `protocolFamily=HTTP` 或 `applicationProtocol=http1` 的报文：

1. 基于原始 payload 字节定位 `\r\n\r\n` 分隔符。
2. 分离起始行、Header、Body。
3. 使用 `Content-Type` 判断是否为 JSON。
4. 仅在 Body 未截断且 JSON 解析成功时生成 `bodyJsonPretty`。

#### 6.4.3 截断策略

如果报文已被采集层截断：

- 仍返回 `textRaw` 与 `hex`；
- `http.bodyDetected` 可为真；
- 但若 JSON 不完整，则 `bodyJsonPretty` 返回空，并通过 `bodyTruncated=true` 明确说明不可格式化原因。

### 6.5 前端实现要求

- 列表页结构基本不变，重点优化详情弹层。
- 新增通用组件概念：
  - `tab-switch`
  - `sub-tab-switch`
  - `content-toolbar`
  - `copy-feedback`
- 复制能力优先使用浏览器剪贴板 API；失败时降级为选中提示。
- 保留“下载负载”能力，避免仅依赖预览。

### 6.6 验收标准

#### 功能验收

- 报文详情只显示一个主体阅读区，可在 `文本` / `十六进制` 间切换。
- HTTP + JSON + 未截断场景下，出现 `JSON 格式化` 入口。
- 点击 `复制当前视图` 后可复制当前正在展示的内容。
- HTTP 报文在能识别 Body 时可使用 `复制请求体`。
- 报文截断或非 JSON 时，格式化入口不误显示，并有明确提示。

#### 体验验收

- 1440px 宽度下，详情弹层打开后，主体阅读区域明显大于当前版本。
- 移动端或窄屏下，Tab、复制按钮、主体区域仍可正常操作。

## 7. 子项 B：全局信息密度优化

### 7.1 设计原则

- 默认进入“紧凑但不压抑”的控制台模式。
- 优先压缩装饰性空间，不压缩关键信息的可读性。
- 所有页面统一收敛字号、控件高度、间距与圆角。
- 不做“仅报文页变紧凑”，而是建立一套全站统一密度规范。

### 7.2 优化范围

覆盖以下页面：

- 登录页
- 运行状态
- 映射配置
- 连接记录
- 报文记录
- 通用弹层

### 7.3 视觉密度规范

建议引入一组统一密度 token，作为默认值：

| 项目 | 当前倾向 | 建议目标 |
| --- | --- | --- |
| 基础正文 | 14px 左右 | 13px |
| 次级说明文字 | 13-15px | 12-13px |
| 大标题 | 28-40px / 32-52px | 22-32px |
| 顶栏高度 | 72px | 56-60px |
| 主按钮高度 | 42px | 34-36px |
| 输入框高度 | 44px | 34-36px |
| 导航项高度 | 54px | 40-44px |
| 表格单元格 padding | 14px | 8-10px |
| 面板内边距 | 20-24px | 14-18px |
| 大圆角 | 24-32px | 16-20px |

### 7.4 具体调整要求

#### 7.4.1 侧边导航

- 压缩 Logo 区与说明文案高度。
- 导航项高度下调，减少上下留白。
- 保留激活态清晰度，但弱化过大的装饰阴影。

#### 7.4.2 顶部栏与 Hero

- 顶部栏改为更轻量的工具栏风格。
- 页面 Hero 保留，但缩小标题与上下 padding。
- 非关键说明文案尽量单行或两行内结束。

#### 7.4.3 表格与筛选区

- 表格行高降低，默认同屏显示更多记录。
- 筛选条输入框、按钮统一紧凑高度。
- 时间、大小、状态类列优先使用更短文案，减少换行。

#### 7.4.4 卡片与弹层

- 卡片圆角与阴影收敛，降低视觉膨胀感。
- 弹层头部高度降低，把可视高度更多留给正文。
- `detail-grid` 改为更高密度的信息块，避免一屏只显示少量字段。

### 7.5 响应式要求

- 桌面端以提升每屏信息量为主。
- 平板与窄屏端允许纵向堆叠，但仍使用紧凑字号和控件高度。
- 移动端不追求展示所有摘要字段，优先保证可操作与可滚动。

### 7.6 实施建议

建议先抽离全局变量，再局部替换样式：

1. 新增字号、间距、圆角、控件高度 token。
2. 先改通用元素：按钮、输入框、表格、panel、modal。
3. 再微调页面级组件：sidebar、topbar、hero、detail-grid、preview 区。

这样可以避免每个页面单独硬改，后续维护成本更低。

### 7.7 验收标准

- 1440px 宽度下：
  - 报文列表同屏可见记录数较当前明显增加；
  - 连接列表同屏可见记录数较当前明显增加；
  - 顶栏、Hero、侧栏不再成为主要空间占用者。
- 1366x768 笔记本屏幕下：
  - 无明显拥挤或文字重叠；
  - 主要按钮与输入框仍可准确点击；
  - 详情弹层主体区可直接阅读，不需要频繁滚动才看到报文内容。

## 8. 风险与约束

### 8.1 技术风险

- 当前预览逻辑会破坏 HTTP 原始文本结构，必须补充后端解析字段，否则 JSON 格式化需求无法稳定达成。
- 若采集时 payload 已截断，则只能做“有限预览”，不能承诺所有 JSON 都可格式化。

### 8.2 体验风险

- 如果只缩小字体、不同步优化间距与布局，页面会变成“字小但仍拥挤”。
- 如果复制语义不清晰，用户会分不清复制的是原文、格式化 JSON，还是十六进制。

## 9. 里程碑建议

### Milestone 1：报文详情能力补齐

- 扩展 `/api/packets/{id}` 返回结构
- 完成 HTTP Body 拆分
- 支持 JSON 格式化与复制

### Milestone 2：全局密度收敛

- 抽取全局密度 token
- 改造表格、筛选区、弹层、顶栏、侧栏
- 回归四个主页面与登录页

## 10. 最终验收口径

本需求完成后，管理端应满足以下结果：

1. 用户在“报文记录”中查看单条报文时，能用单主体 Tab 方式在文本和十六进制间切换。
2. 对明确 HTTP 且请求体为 JSON 的报文，系统能提供稳定的 JSON 格式化视图。
3. 用户可以快速复制当前视图内容，且复制语义清楚。
4. 全站页面在常见办公屏幕上能承载更多有效信息，视觉上更像排障控制台，而不是展示型大卡片页面。
